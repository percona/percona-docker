# The Problem

Standard Percona Server Docker images carry significant attack surface that production deployments shouldn't tolerate.

8 HIGH CVEs

Naive DHI build: bash, ncurses, libreadline, libsystemd, Go stdlib — all flagged

~700 MB image

MySQL Shell (Python + V8), mysql client, mysqldump, mysql_upgrade — rarely needed at runtime

bash in runtime

Full interactive shell with readline → libtinfo → ncurses dependency chain and 4 HIGH CVEs

libsystemd

mysqld links sd_notify() — never used in containers but CVE-2026-29111 rides along

root-capable

apt, dpkg, curl, wget all present — one RCE away from package-manager pivoting

~50 users in /etc/passwd

nobody, sshd, systemd-*, messagebus, _apt — none needed by mysqld

# Security Features

◆ Non-root runtime

USER mysql (uid 1001), no setuid binaries, cap-drop ALL compatible

◆ No bash / no readline

POSIX dash only. Kills libreadline → libtinfo → ncurses CVE chain (4 HIGHs)

◆ Stubbed libsystemd

30-line C stub compiled in build. sd_notify → return 0. Kills CVE-2026-29111

◆ ldd-pruned libs

Only shared libraries mysqld + plugins actually link. No unused Debian bloat

◆ 2-line /etc/passwd

Only root + mysql. No nobody, sshd, systemd-*, messagebus, _apt, daemon

◆ Inline SBOM

SPDX 2.3 at /usr/local/percona-server.spdx.json. DHI base is Docker-signed

+ Supports --read-only rootfs · + no-new-privileges · + Full plugin set preserved (InnoDB, RocksDB, audit_log, keyring_vault)
